Open-Closed Principle (OCP)	Software artifacts should be open for extension but closed for modification — adding new functionality should only require adding new code, not changing existing code. [Guideline 5]	vocab guideline5
Customization point	A place in existing code intentionally designed to let you plug in your own behavior without modifying that existing code. [Guideline 5]	vocab guideline5
What goes wrong when you design an enum to identify subtypes (e.g. DocumentType)?	All subtypes are artificially coupled — adding a new type forces a change to the enum, which triggers recompilation of every class that depends on it. That's an OCP violation. [Guideline 5]	why-it-matters guideline5
What is the key difference between OCP and SRP?	SRP is about separating things that change for different reasons. OCP is about being conscious of *how* code will be extended and designing explicitly for that. They often overlap but OCP has a more explicit architectural intent. [Guideline 5]	why-it-matters guideline5
A PDF and Word class both depend on a DocumentType enum to identify themselves during serialization. You want to add XML. What's wrong and how do you fix it?	OCP violation — adding XML forces a change to DocumentType, which causes PDF and Word to recompile even though they're unrelated to XML. Fix: move serialization logic into a separate Serialization component on a lower architectural level that depends on the document types, not the other way around. [Guideline 5]	scenario guideline5
A Shape base class has a ShapeType enum used in a drawAllShapes() switch statement. You add a Triangle. What's wrong?	OCP violation — adding Triangle requires modifying the enum and every switch statement that uses it across the codebase. Any class depending on ShapeType recompiles. Fix: use virtual dispatch — let each shape know how to draw itself, so adding Triangle only requires adding the new class. [Guideline 5]	scenario guideline5
You want to provide a custom swap for your type MyConfig. How do you do it correctly so generic code picks it up?	Define a free swap() in the same namespace as MyConfig. At the call site, use `using std::swap;` followed by an unqualified `swap(a, b)` — ADL will find your custom version via the namespace of the arguments. [Guideline 5]	scenario guideline5
A team adds a new Payment type to a PaymentType enum shared across the codebase. All existing payment classes recompile. What principle is violated and what's the fix?	OCP violation — the enum artificially couples all payment types. Fix: move the enum into a lower-level component (e.g. PaymentSerializer) that depends on the payment types, so existing types are unaware of new additions. [Guideline 5]	scenario guideline5
You need a hash function for your custom Key type to use it in std::unordered_map. How do you do it without modifying any existing code?	Specialize std::hash<Key> — template specialization is a customization point that lets you inject new behavior without touching existing code. [Guideline 5]	scenario guideline5
A codebase has a Processor class with a processCSV(), processJSON(), and processXML() method. A new format comes in — you have to add a method and recompile Processor. What's wrong?	OCP violation — Processor is not closed for modification. Fix: introduce an abstraction (e.g. a Format interface or a template) so new formats can be added as separate classes without touching Processor. [Guideline 5]	scenario guideline5
