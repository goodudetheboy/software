Overload set	A group of free functions that share the same name — together they form a compile-time abstraction where the compiler picks the right function based on the argument types. [Guideline 8]	vocab guideline8
Free function	A function that is not a member of any class. It can be added to any type non-intrusively, without modifying the type itself. [Guideline 8]	vocab guideline8
Intrusive vs non-intrusive extension	Adding a member function is intrusive — you must modify the class. Adding a free function is non-intrusive — you extend behavior without touching the existing type. [Guideline 8]	vocab guideline8
Why does joining an overload set immediately subject you to the LSP?	The overload set name carries implicit semantic expectations. Any function you add under that name must fulfill those expectations — violating them breaks callers who rely on the assumed behavior, just like a bad subclass breaks LSP. [Guideline 8]	why-it-matters guideline8
Why are free functions preferable to member functions from a design perspective?	Free functions have no implicit `this` pointer — they don't couple to a class. They become isolated, reusable services that reduce dependencies, improve testability, and can be applied to types you don't own. [Guideline 8]	why-it-matters guideline8
You write swap() for your Widget but only swap the first data member, not the second. A coworker calls swap() and spends an hour debugging incorrect state. What principle is violated?	LSP violation on the overload set — swap() carries the expectation that the *entire* observable state is swapped. Swapping only part of the object breaks that contract silently at compile time. Fix: swap all data members. [Guideline 8]	scenario guideline8
A traverseRange() function calls range.begin() and range.end() as member functions. You try to pass a raw C array and it fails to compile. What's wrong and how do you fix it?	The function creates an artificial requirement — it forces Range to have member begin()/end(). Fix: use free begin()/end() with `using std::begin; using std::end;` so ADL can find overloads for any type, including raw arrays. [Guideline 8]	scenario guideline8
A developer names their binary search function find(). What's wrong?	Naming violation — find() carries the semantic expectation of a *linear* search over an unsorted range. Using it for binary search violates those expectations and misleads callers who don't know the range must be sorted. Use a name like binary_search() instead. [Guideline 8]	scenario guideline8
A team adds begin() and end() free functions to their EventLoop class — begin() starts the loop, end() stops it. What's wrong?	LSP violation on the overload set — begin() and end() are expected to return iterators for range traversal, not to control process lifecycle. This violates naming conventions and breaks any generic code that calls begin()/end() expecting iterators. [Guideline 8]	scenario guideline8
Your codebase has a serialize() free function overloaded for several types. A new developer adds a serialize() for their Config type but it writes to a file instead of returning bytes like every other overload. What's the risk?	Silent LSP violation on the overload set — the name serialize() implies a consistent contract (returning serialized data). The new overload violates that contract. Generic code that calls serialize() and uses the return value will break or behave incorrectly. [Guideline 8]	scenario guideline8
A logging library wants to add log() support for a third-party type it doesn't own. Should it use a member function or a free function?	Free function — member functions are intrusive and require modifying the third-party type, which is impossible. A free function can be added non-intrusively in any namespace, extending behavior without touching existing code. [Guideline 8]	scenario guideline8
