Liskov Substitution Principle (LSP)	The "L" in SOLID. If φ(x) holds for objects of type T, then φ(y) must hold for objects of subtype S. In practice: a subtype must honor all behavioral expectations of its base type. [Guideline 6]	vocab guideline6
Behavioral subtyping	The idea that a subtype must not just be structurally compatible with its base, but must also preserve its expected behavior — anywhere you use a base type, you can swap in a subtype without surprises. [Guideline 6]	vocab guideline6
Covariant return type	A subtype's overriding function may return a more derived type than the base declared. Safe because callers expecting the base return type still get something compatible. Supported natively in C++. [Guideline 6]	vocab guideline6
Contravariant function parameter	A subtype's overriding function may accept a broader (super) type than the base declared — making it more permissive, never less. Theoretically correct per LSP, but NOT supported in C++. [Guideline 6]	vocab guideline6
What goes wrong when a subtype strengthens a precondition?	Callers of the base type pass inputs that were valid for the base but now get rejected by the subtype — the subtype breaks the contract. LSP violation. [Guideline 6]	why-it-matters guideline6
What goes wrong when a subtype weakens a postcondition?	Callers expect the guarantees the base promised on exit. The subtype delivers less — callers can no longer rely on the abstraction. LSP violation. [Guideline 6]	why-it-matters guideline6
A Rectangle base class has setWidth() and setHeight() that work independently. Square overrides both to keep all sides equal. A function sets width to 7 and height to 4, then asserts area == 28. What happens and why?	The assert fails. Square's invariant (all sides equal) forces setHeight() to also change the width, violating the caller's expectation that width and height are independent. LSP violation — Square is not a behavioral subtype of Rectangle. [Guideline 6]	scenario guideline6
Base::f() accepts all i > 0. Derived::f() overrides it and asserts i > 10. What's wrong?	Precondition strengthened — inputs between 1 and 10 were valid for Base but are now rejected by Derived. Callers of Base are surprised. LSP violation. [Guideline 6]	scenario guideline6
Base::f() guarantees it returns a value > 0. Derived::f() overrides it and returns any int. What's wrong?	Postcondition weakened — callers relying on the guarantee that the result is positive will get unexpected values from Derived. LSP violation. [Guideline 6]	scenario guideline6
X has a protected value_ with invariant: must stay in [1..10]. Y (subtype of X) sets value_ = 11 in its constructor. What's wrong?	Y breaks X's invariant immediately after construction. Any code relying on value_ being in range will behave incorrectly. LSP violation — subtypes must preserve supertype invariants. [Guideline 6]	scenario guideline6
A function f(Base& b) checks dynamic_cast<Special*>(&b) to handle Special differently from other derived types. What's wrong with this pattern?	It's an LSP workaround — f() now depends on the misbehavior of a specific derived type rather than the abstraction. This is the exact anti-pattern LSP violations produce. [Guideline 6]	scenario guideline6
A PaymentProcessor base class has process(amount) that accepts any positive amount. A derived CorporateProcessor overrides it to reject amounts under $1000. Is this an LSP violation?	Yes — precondition strengthened. Callers passing small amounts that were valid for PaymentProcessor will be incorrectly rejected by CorporateProcessor. [Guideline 6]	scenario guideline6
A Logger base class guarantees log() always writes to disk before returning. A derived AsyncLogger overrides log() to queue the write and return immediately. Is this an LSP violation?	Yes — postcondition weakened. Callers relying on the guarantee that the write is complete after log() returns will be surprised by AsyncLogger's deferred behavior. [Guideline 6]	scenario guideline6
